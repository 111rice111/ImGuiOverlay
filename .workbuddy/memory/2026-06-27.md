## v1.5 修复：g_all_maps 索引越界导致显示错误地图

### 根因分析
`LoadMapConfigFromJSON()` 中，每个 JSON map entry 创建独立 slot（g_all_maps.push_back）。偶数 slot 有 2 层（含 auto-created floor 1），奇数 slot 只有 1 层。但 `TryAutoDetectMap` 中多处直接用 `GetFloorFromPlayerZ(Z)` 返回 0/1 设置 `g_current_floor_index`，导致奇数 slot 访问 `[1]` 时越界，读取相邻 slot 数据。

### 修复方案
新增 `SafeClampFloorIdx(int mapIdx, int floorIdx)` 函数：
- 检查 `floors.size() - 1`，大于时 clamp 到最大值
- 在所有访问 `g_all_maps` 前做钳制（5 处）

## v2.3 修复首次地图识别为"地图1" (13:46)

### 用户反馈
测试时明明不在地图1，但系统仍然识别为"地图1"

### 根因分析
`g_current_map_index` 全局初始化 = 0（line 316），而 g_all_maps[0] 始终是"地图1 一楼"。
LOCKED 状态的评分路径只有重检（60帧间隔 + 15分差距）才会触发 ExecuteMapSwitch，
所以首次进入游戏时，系统从未执行地图切换流程，一直显示 g_all_maps[0].name = "地图1"。

额外因素：
- config 持久化恢复（line 822）也会把跨局残留的旧地图索引恢复回来
- ResetMapState() 定义了但从未被调用（不过 g_need_reinit 的 reset 块有类似逻辑）

### 修复（3 处修改）
1. g_current_map_index 默认值: 0 → -1
2. 移除 config 持久化恢复（不再从 overlay_config.txt 恢复旧地图索引）
3. LOCKED 状态新增首次检测分支：g_current_map_index == -1 时若评分 ≥ 60 分且无并列，调用 ExecuteMapSwitch

## v2.4 修复一楼误判为二楼 (16:50)

### 用户反馈
"我现在在1楼，他给我判断的是在2楼"

### 根因
`GetFloorFromPlayerZ` 使用硬编码阈值 `Z > 190` 判定 2 楼。
部分游戏地图 1 楼的海拔本身就超过 190，导致玩家永远被判为 2 楼。

### 修复
`GetFloorFromPlayerZ` 改为读取当前地图的 `floorZThreshold`（来自 JSON 的 `floor_z_threshold` 字段）：
```cpp
float threshold = 250.0f;
if (g_current_map_index >= 0 && ...)
    threshold = g_all_maps[g_current_map_index][0].floorZThreshold;
return (pos.Z > threshold) ? 1 : 0;
```
